# Djinn Developer Guide & Deep Code Review

Welcome to the internal documentation for **Djinn**. This guide provides a deep dive into the architecture, execution flows, and extension patterns required to understand, use, and scale the bot effectively.

---

## 1. The Core Architecture

Djinn operates on a modular, event-driven architecture built on top of `discord.py`. Unlike traditional bots that parse static command prefixes (like `!ban`), Djinn is an **Agentic Bot**. It intercepts natural language, routes it through an LLM, and executes structured tool calls.

### Key Components:
- **`main.py`**: The entry point. It sets up logging, initializes the SQLite database (`utils/database.py`), spins up the internal HTTP API (`utils/api_server.py`), and loads all cogs dynamically.
- **`cogs/nlp_handler.py`**: The Discord event listener that captures user messages and mentions, forwarding them to the Orchestrator.
- **`utils/orchestrator.py`**: The brain of the bot. It prepares the prompt, enforces security restrictions based on user roles, filters tools contextually, and calls the LLM via `LLMClient`.
- **`utils/discord_tools.py`**: The execution engine (`ToolExecutor`). It maps the LLM's JSON tool calls to actual asynchronous Python methods.
- **`utils/security.py`**: The zero-trust gatekeeper. Validates permissions (RBAC) and prevents Server-Side Request Forgery (SSRF) during web requests.

## 2. The Semantic Orchestrator (`utils/orchestrator.py`)

The Orchestrator is responsible for minimizing token usage while maximizing the bot's contextual awareness. It achieves this through **Semantic Tool Routing**.

### How it works:
1. **Message Interception**: When a message arrives, `_prepare_content()` replaces Discord IDs (`<@123>`) with readable names (`User (ID: 123)`), giving the LLM immediate context without needing to execute a lookup tool.
2. **Intent Analysis**: The Orchestrator does a fast keyword and regex pass over the user's prompt. 
3. **Dynamic Injection**: Instead of injecting all 137 tools, it only loads the relevant categories. For example:
   - Mentions of "ban", "kick", or "spam" inject the `moderation` toolset.
   - Mentions of "credits", "debt", or "banco" inject the `economy` toolset.
   - The `core` tools (like `send_message`, `search_messages`) are always included.
4. **Pipeline Routing**: 
   - If the user is in the `youkai_readers` database list, they go through the **Admin Pipeline** (full access).
   - If not, they are routed to the **Public Pipeline** (`_process_as_user`), which strips all dangerous administrative tools from the prompt, physically preventing the LLM from executing them regardless of prompt injection attempts.

## 3. Extending Djinn: How to Add a New Tool

Djinn uses a strict contract between the LLM schemas and the Python execution handlers.

### Step 1: Declare the Tool Schema
Open `utils/tools/_declarations.py` and add your tool to `TOOL_DECLARATIONS` using the `_decl` helper:
```python
_decl("kick_user",
    "Expulsa a un usuario del servidor.",
    {
        "user_id": _str("ID numérico del usuario"),
        "reason": _str("Razón de la expulsión")
    },
    ["user_id"] # Required parameters
)
```

### Step 2: Implement the Handler
Open `utils/discord_tools.py`, locate the `ToolExecutor` class, and add an asynchronous method matching the pattern `_do_<tool_name>`:
```python
async def _do_kick_user(self, user_id: str, reason: str = "Sin razón", **_):
    member = self.guild.get_member(int(user_id))
    if not member:
        return {"error": "Usuario no encontrado"}
    
    await member.kick(reason=reason)
    return {"success": True, "message": f"Usuario {member.name} expulsado."}
```
*Note: Always include `**_` in the signature to absorb unexpected kwargs generated by LLM hallucinations.*

### Step 3: Register the Permission (If Administrative)
Open `utils/security.py` and add your tool to `TOOL_REQUIRED_PERMS` to enforce Discord role checks:
```python
TOOL_REQUIRED_PERMS = {
    "kick_user": discord.Permissions(kick_members=True),
    # ...
}
```

## 4. Subsystems Deep Dive

### Trust-First Moderation (Automod v3)
Located in `cogs/automod_v3.py`. Instead of banning users instantly for minor infractions, Djinn uses the **Goodfaith** engine. It assigns a "Trust Score" to users based on their account age, message frequency, and past infractions. Actions are graduated (Warn → Timeout → Ban) based on this dynamic score.

### MediaGuard (Visual Perception)
Located in `cogs/media_guard/`. Uses an ONNX-compiled MobileNetV3 model to generate feature vectors (embeddings) of images uploaded to Discord. It cross-references these vectors against an `hnswlib` index (`data/banned_media.bin`) using cosine similarity to instantly detect and delete NSFW, prohibited, or spam imagery.

### The Loan Shark Economy
Located in `cogs/loan_shark.py` and `cogs/treasury.py`. A fully transactional SQLite-backed economy. Users can request loans which accrue dynamic interest based on global server liquidity. Handled meticulously to prevent race conditions using SQL `WAL` mode and row-level locks.

## 5. Security & SSRF Protection
Djinn features native tools for web scraping (`read_url`, `screenshot_url`). To prevent malicious admins from asking the bot to scrape internal AWS metadata or local files, `utils/security.py` implements a strict SSRF guard. It resolves DNS records asynchronously and blocks any requests mapping to private CIDR ranges (e.g., `10.0.0.0/8`, `127.0.0.0/8`, `169.254.169.254`).
